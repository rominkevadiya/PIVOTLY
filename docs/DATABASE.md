# Database Architecture: Venture Intelligence Platform

## Database Overview

The Pivotly backend relies on a PostgreSQL database for persistence. The database schema is defined and managed using SQLAlchemy declarative ORM models and Alembic for schema migrations.

The database tracks three primary entities:
1.  **Users:** Stores identity, authentication credentials, and account status.
2.  **Reports:** Stores the raw text of user submissions alongside the structured JSON output generated by the AI model.
3.  **Rate Limits:** Tracks daily action quotas per user to prevent abuse of the AI API.

## Table Documentation

### 1. `users` Table
**Purpose:** Persists registered user identities and credentials.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | Generated automatically via `uuid.uuid4`. |
| `email` | String(255) | Unique, Indexed, Not Null | User's email address for login. |
| `full_name` | String(255) | Not Null | User's display name. |
| `hashed_password` | String(255) | Not Null | Bcrypt hash of the user's password. |
| `is_active` | Boolean | Not Null, Default `True` | Soft deletion/ban flag. |
| `created_at` | DateTime | Not Null, Default `UTC NOW` | Timestamp of account creation. |
| `updated_at` | DateTime | Not Null, Default `UTC NOW` | Auto-updates on row modification. |

**Relationships:**
*   One-to-Many to `reports.user_id`
*   One-to-Many to `rate_limits.user_id`

### 2. `reports` Table
**Purpose:** Stores user submissions and the generated AI analysis reports.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | Generated automatically via `uuid.uuid4`. |
| `user_id` | UUID | Foreign Key (`users.id`), Indexed, Nullable | The user who generated the report. |
| `idea_text` | Text | Not Null | The raw startup idea provided by the user. |
| `report_json` | JSONB | Nullable | The complex, validated AI payload (Pydantic schema). `NULL` until `COMPLETED`. |
| `status` | String(50) | Not Null, Default `PENDING` | Report lifecycle state: `PENDING`, `SCRAPING`, `GENERATING`, `COMPLETED`, `FAILED`. |
| `schema_version` | Integer | Not Null, Default `2` | Enables V1/V2 backward compatibility routing via `ReportValidationFactory`. |
| `industry` | String(255) | Nullable | Extracted from AI output on completion for quick filtering. |
| `market_potential`| String(50) | Nullable | High/Medium/Low rating set on completion. |
| `recommendation` | String(50) | Nullable | Build/Pivot/etc. set on completion. |
| `error_message` | Text | Nullable | Captured error string if `status = FAILED`. |
| `created_at` | DateTime | Not Null, Default `UTC NOW` | Timestamp of report row creation. |

**Constraints:**
*   `user_id` relies on the existence of the user, though it is marked `Nullable` to potentially support guest analysis in the future.

### 3. `rate_limits` Table
**Purpose:** Tracks daily quota usage for specific actions (e.g., "idea_submission").

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | Integer | Primary Key, Auto-increment | Internal numeric identifier. |
| `user_id` | UUID | Foreign Key (`users.id`), Indexed, Not Null | The restricted user. |
| `action` | String(50) | Not Null | Identifier of the action (e.g., 'idea_submission'). |
| `window_date` | Date | Not Null | The calendar day the limit applies to. |
| `count` | Integer | Not Null, Default `0` | Number of actions performed in the window. |
| `updated_at` | DateTime | Not Null, Default `UTC NOW` | Auto-updates on row modification. |

**Constraints:**
*   `ON DELETE CASCADE` on `user_id` (If a user is deleted, their rate limit history is purged).
*   **Unique Constraint:** `uq_user_action_date` on `(user_id, action, window_date)`. This is vital for allowing atomic Postgres upserts (`ON CONFLICT`).

## Query Patterns

The application interacts with the database in distinct, predictable patterns:

1.  **Authentication (Read-Heavy):**
    *   `SELECT ... FROM users WHERE email = ?` (Triggered on every login).
    *   `SELECT ... FROM users WHERE id = ?` (Triggered on every protected route to validate the JWT `sub` claim).
2.  **Dashboard Load (Read-Heavy, Paginated):**
    *   `SELECT ... FROM reports WHERE user_id = ? ORDER BY created_at DESC LIMIT ? OFFSET ?` (Triggered on `/api/v1/reports`).
    *   *Note:* The query only pulls top-level columns mapped to Pydantic summaries, though SQLAlchemy fetches the whole row (including `JSONB`) by default.
3.  **Report Generation (Write-Heavy, Transactional):**
    *   **Check:** `SELECT count FROM rate_limits WHERE user_id = ? AND action = ? AND window_date = ?`
    *   **Insert:** `INSERT INTO reports (...) VALUES (...)`
    *   **Atomic Upsert:** `INSERT INTO rate_limits (user_id, action, window_date, count) VALUES (?, ?, ?, 1) ON CONFLICT (user_id, action, window_date) DO UPDATE SET count = count + 1`

## Performance Considerations

### Current Indexing
*   **Primary Keys:** Standard B-Tree indexes on `users.id`, `reports.id`, and `rate_limits.id`.
*   **Foreign Keys & Lookups:** Explicit B-Tree indexes exist on `users.email` (for fast login lookups), `reports.user_id` (for fast dashboard history fetching), and `rate_limits.user_id`.
*   **Unique Constraints:** The unique constraint on `rate_limits(user_id, action, window_date)` acts as a composite index, speeding up the daily quota checks.

### Optimization Notes
*   **`JSONB` Column:** The `reports.report_json` column utilizes PostgreSQL's binary JSON format. This avoids parsing overhead upon reading. Currently, the application does not execute SQL queries against fields *inside* the JSON. If searching within reports is needed later (e.g., finding all reports where the competitor is "Stripe"), a `GIN` index on `report_json` will be necessary.
*   **Atomic Rate Limiting:** By using the database for rate limiting with an `ON CONFLICT` clause, the app avoids race conditions when multiple requests fire simultaneously, without needing external infrastructure like Redis.

## Future Expansion (Realistic Schema Improvements)

1.  **JSONB Indexing:** As the platform grows, users will want to search their past reports. Adding a `GIN` index to `reports.report_json` or extracting highly queried fields into dedicated relational columns will be necessary.
2.  **Soft Deletion for Reports:** Implement an `is_deleted` boolean on the `reports` table. Currently, reports cannot be deleted by users.
3.  **Deferred Column Loading:** In the `ReportRepository`, the `get_by_user_id` method fetches the heavy `report_json` payload for every item in the history list. Using SQLAlchemy's `defer('report_json')` in this specific query will significantly reduce network bandwidth and memory overhead.

## JSONB Payload Structure (`VentureReportV2`)

The `report_json` column stores the complex validation schemas generated by the AI orchestration pipeline. The column is `NULL` until the background task completes (`status = COMPLETED`).

```python
class VentureReportV2(BaseModel):
    # Input summary
    idea_summary: str

    # Agent Outputs (each can be SectionError if agent fails)
    research_context: ResearchContext | SectionError | None
    competitor_analysis: CompetitorAnalysis | SectionError | None
    moat_analysis: MoatAnalysis | SectionError | None
    contrarian_analysis: ContrarianAnalysis | SectionError | None
    action_plan: ActionPlan | SectionError | None

    # Deterministic Scoring (always present if any agents succeeded)
    scoring_rubric: ScoringRubric

    # Structured Recommendation
    recommendation: RecommendationSection

class RecommendationSection(BaseModel):
    decision: str          # "Build", "Pivot", "Research Further"
    confidence: str        # "High", "Medium", "Low"
    confidence_score: int  # 0-100 (maps to overall_score)
    evidence: str
    rationale: str
```

*(Note: Every agent-specific section leverages a Union Type allowing `SectionError` to handle partial pipeline failures gracefully without corrupting the database record.)*
